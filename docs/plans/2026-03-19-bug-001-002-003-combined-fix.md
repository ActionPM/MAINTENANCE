## Diagnosis Summary

### Cluster 1: confidence-coverage (BUG-001 + BUG-003)

The same root cause produces both bugs:

`classification_cues.json` → `Category.maintenance` keywords are `[leak, broken, repair, not working, clog]`. This list misses:
- HVAC terms: heat, cold, freezing, heater, hvac, radiator, thermostat, ac, furnace
- Plumbing overflow terms: overflow, overflowing, flooding, backed up
- General maintenance terms: toilet, faucet, sink, pipe, drain, mold, pest, cockroach

For any text that doesn't contain one of those 5 keywords, `cue_strength` for Category = 0.0. The confidence formula maxes at `0.40*0 + 0.25*1.0 + 0.20*0.95 = 0.44` — below `medium_threshold` (0.65). This forces a "Is this maintenance?" follow-up question for issues that are unambiguously maintenance.

Additional gaps:
- `Location.suite` cues miss possessive patterns ("my toilet", "my heater") that strongly imply suite
- `Sub_Location.general` cues miss heat/cold/whole-unit terms
- Taxonomy lacks `entire_unit` and `multiple_rooms` Sub_Location values
- Follow-up prompt doesn't instruct Claude to include whole-unit options for HVAC
- UI renders raw taxonomy slugs (`common_living_dining`) instead of display labels
- Gold examples `gold-016` and `gold-019` expect zero followup fields, contradicting runtime

### Cluster 2: demo-ux (BUG-002)

`INPUT_STATES` in `chat-shell.tsx` is `new Set(['intake_started', 'unit_selected'])`. For multi-unit users (Bob), `CREATE_CONVERSATION` returns `intake_started`. Since `intake_started` is in `INPUT_STATES`, the `MessageInput` renders alongside the `UnitSelector`. The demo `defaultValue` pre-fills the message, making it trivial to Send before selecting a unit. The backend guard catches it (`UNIT_NOT_RESOLVED`), but the error response isn't handled gracefully in the UI.

```typescript
import labelsJson from '../taxonomy-labels.json' with { type: 'json' };

const labels: Record<string, Record<string, string>> = labelsJson.labels;

/**
 * Look up a display-friendly label for a taxonomy slug.
 * Returns the label if mapped, otherwise formats the slug
 * (underscores → spaces, first letter capitalized).
 */
export function getTaxonomyLabel(field: string, slug: string): string {
  const fieldLabels = labels[field];
  if (fieldLabels && fieldLabels[slug]) {
    return fieldLabels[slug];
  }
  return slug
    .replace(/_/g, ' ')
    .replace(/^\w/, (c) => c.toUpperCase());
}
```

## Batch 3: UI Fix (BUG-002)

> **Purpose**: Prevent message submission before unit selection.

**Critical context**: `handleCreateConversation()` in `packages/core/src/orchestrator/action-handlers/create-conversation.ts` returns `ConversationState.INTAKE_STARTED` for ALL tenants — both single-unit and multi-unit. It does NOT auto-select a unit. The `SELECT_UNIT` guard in `guards.ts` handles auto-selection (single unit → `unit_selected`, multiple → `unit_selection_required`), but that guard is only invoked when `SELECT_UNIT` is dispatched as a separate action.

Therefore, simply removing `intake_started` from `INPUT_STATES` would dead-end BOTH single-unit and multi-unit users — neither the UnitSelector (gated on `unit_selection_required`) nor the MessageInput (gated on `unit_selected`) would render.

### Task 3.1: Change CREATE_CONVERSATION to auto-resolve unit for single-unit tenants

**File**: `packages/core/src/orchestrator/action-handlers/create-conversation.ts`

For single-unit tenants, the handler should resolve the unit (via `deps.unitResolver`), set all three session fields (`setSessionUnit` + `setSessionScope` + `setBuildingId`), and return `unit_selected`. For multi-unit tenants, return `unit_selection_required` so the UnitSelector renders and the MessageInput does not.

**Critical**: Use the same resolution pattern as `handleSelectUnit` in `select-unit.ts`:
- Call `deps.unitResolver.resolve(unitId)`
- If it returns `null`, do NOT advance to `unit_selected` — return `unit_selection_required` with a retry path (see code block and rationale below)
- If it returns a valid result, call `setSessionUnit(session, unitId)`, `setSessionScope(updatedSession, { property_id, client_id })`, `setBuildingId(updatedSession, building_id)`

The session helpers have these exact signatures (from `session.ts`):
- `setSessionUnit(session, unitId: string)` — sets `unit_id`
- `setSessionScope(session, { property_id, client_id })` — sets scope
- `setBuildingId(session, buildingId: string)` — sets building

```typescript
// New logic (replaces the single return statement):
if (unitCount === 1) {
  const autoUnitId = request.auth_context.authorized_unit_ids[0];
  const unitInfo = await deps.unitResolver.resolve(autoUnitId);
  if (!unitInfo) {
    // Resolver failed — fall to unit_selection_required so the UnitSelector renders.
    // The tenant can retry via SELECT_UNIT, which has its own resolver-null rejection
    // with a visible error path. Do NOT fall back to intake_started — Task 3.2 removes
    // it from INPUT_STATES, so it would be a silent dead-end (no MessageInput, no
    // UnitSelector, and the 201 response with errors is not surfaced by the web client).
    return {
      newState: ConversationState.UNIT_SELECTION_REQUIRED,
      session,
      uiMessages: [
        { role: 'agent', content: 'Welcome! Please select your unit to get started.' },
      ],
      quickReplies: [{
        label: `Unit ${autoUnitId}`,
        value: autoUnitId,
        action_type: ActionType.SELECT_UNIT,
      }],
      errors: [{ code: 'UNIT_RESOLVE_FAILED', message: 'Auto-resolution failed, manual selection required' }],
      eventPayload: { authorized_unit_ids: request.auth_context.authorized_unit_ids, auto_resolve_failed: true },
    };
  }
  let updatedSession = setSessionUnit(session, autoUnitId);
  updatedSession = setSessionScope(updatedSession, {
    property_id: unitInfo.property_id,
    client_id: unitInfo.client_id,
  });
  updatedSession = setBuildingId(updatedSession, unitInfo.building_id);
  return {
    newState: ConversationState.UNIT_SELECTED,
    session: updatedSession,
    uiMessages: [{ role: 'agent', content: 'Welcome! How can we help you today?' }],
    eventPayload: { authorized_unit_ids: request.auth_context.authorized_unit_ids, auto_selected_unit: autoUnitId },
  };
} else {
  return {
    newState: ConversationState.UNIT_SELECTION_REQUIRED,
    session,
    uiMessages: [{ role: 'agent', content: 'Welcome! Please select which unit this request is for.' }],
    quickReplies: request.auth_context.authorized_unit_ids.map((id) => ({
      label: `Unit ${id}`,
      value: id,
      action_type: ActionType.SELECT_UNIT,
    })),
    eventPayload: { authorized_unit_ids: request.auth_context.authorized_unit_ids },
  };
}
```

Import `setSessionUnit`, `setSessionScope`, `setBuildingId` from `../../session/session.js`.

**Why `unit_selection_required` and not `intake_started`**: The route returns 201 even when `response.errors` is populated, and the web client doesn't surface `response.errors` as `conv.error`. If the handler returned `intake_started` while Task 3.2 removes it from `INPUT_STATES`, the tenant would see no input, no selector, and no error — a silent dead-end. Falling to `unit_selection_required` renders the UnitSelector with a retry path through the existing `handleSelectUnit` (which has its own resolver-null rejection with a visible UI error).

**Acceptance**: After CREATE_CONVERSATION:
- Alice (1 unit, resolver succeeds) → state = `unit_selected`, unit_id + property_id + client_id + building_id set
- Alice (1 unit, resolver returns null) → state = `unit_selection_required`, UnitSelector with single unit shown, tenant can retry SELECT_UNIT
- Bob (3 units) → state = `unit_selection_required`, UnitSelector visible, MessageInput NOT visible

### Task 3.3: Update startWithQueuedText for new create-conversation contract

**File**: `apps/web/src/hooks/use-conversation.ts`

The `startWithQueuedText` method (line ~120) currently does `createConversation → selectUnit → submitInitialMessage`. After Task 3.1, single-unit tenants land in `unit_selected` after create — and the transition matrix does NOT allow `SELECT_UNIT` from `unit_selected`. So the `selectUnit` step will fail with an invalid transition.

Fix: check the response state after each step. If `createConversation` already returned `unit_selected`, skip `selectUnit`. If `selectUnit` didn't land in `unit_selected` (e.g., resolver-null returned `unit_selection_required` with errors), stop and surface that response instead of calling `submitInitialMessage` on an invalid state.

```typescript
const startWithQueuedText = useCallback(
  async (messages: readonly string[], unitId: string) => {
    setStatus('loading');
    setError(null);
    try {
      const created = await api.createConversation(token);
      const newId = created.conversation_snapshot.conversation_id;
      let current = created;

      // If CREATE_CONVERSATION already auto-selected the unit (single-unit tenant),
      // skip SELECT_UNIT — the transition matrix doesn't allow it from unit_selected.
      if (current.conversation_snapshot.state !== 'unit_selected') {
        current = await api.selectUnit(token, newId, unitId);
      }

      // Guard: only continue to submit if we actually reached unit_selected.
      // SELECT_UNIT can return unit_selection_required with errors (resolver-null)
      // and the route returns 200, not 4xx — so the API client won't throw.
      if (current.conversation_snapshot.state !== 'unit_selected') {
        setResponse(current);
        setStatus('ready');
        return;
      }

      const result = await api.submitInitialMessage(token, newId, messages.join('\n'));
      setResponse(result);
      setStatus('ready');
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Unknown error');
      setStatus('error');
    }
  },
  [token],
);
```

**Acceptance**: "Continue with new issue" flow works for single-unit (skips SELECT_UNIT), multi-unit (calls SELECT_UNIT then submits), and resolver-failure (stops at unit selection, surfaces the response — no invalid transition attempted).

If resolver returns null, handler falls to `unit_selection_required` with a retry path — NOT `intake_started`, which would be a silent dead-end (201 + errors not surfaced by web client). Task 3.3 updates `startWithQueuedText` to skip SELECT_UNIT when already `unit_selected`. Task 3.4 adds tests for all three paths (success, resolver-null, multi-unit). |
| Follow-up prompt Rule 14 causes over-inclusion of options | Rule is scoped to "heating, cooling, or pest issues" only, not all Sub_Location questions. |

## What This Does NOT Change

- No changes to the confidence formula, weights, or thresholds
- No changes to the transition matrix or dispatcher (the matrix already allows `intake_started → SELECT_UNIT → [unit_selected, unit_selection_required]`; we change which state the handler returns, not the matrix itself)
- No changes to the follow-up answer contract (`string | boolean`) — no multi-select
- No changes to auth, rate limiting, or security headers

## What This DOES Change (beyond the confidence cluster)

- `handleCreateConversation` now returns `unit_selected` or `unit_selection_required` instead of always `intake_started`. This changes the post-creation landing state, which affects all tenants, not just the demo flow. The `intake_started` state remains in the transition matrix for RESUME edge cases but is no longer the initial landing state for new conversations.
