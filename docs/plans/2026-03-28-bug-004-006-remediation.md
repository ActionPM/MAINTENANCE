# Plan: Targeted Remediation for BUG-004 and BUG-006

## Summary

BUG-004 (P2) is a follow-up reclassification defect: `answer-followups.ts:152` computes cue scores from `issue.summary + issue.raw_excerpt` only, so tenant follow-up answers never improve cue-based confidence and Category stays below the 0.70 gating threshold — causing redundant "maintenance or management?" re-asks. BUG-006 (P3) is a confirmation presentation defect: `payload-builder.ts` copies the full classification map including cross-domain `not_applicable` values, and `confirmation-panel.tsx` renders them as unlabeled chips with no field context.

This plan stays intentionally narrow. It does not roll BUG-004 into the broader BUG-005/BUG-007 confidence-policy work, and it does not redesign the confirmation flow into freeform prose.

## Tasks

### Task 1: Add `buildEnrichedCueText` helper to cue-scoring.ts

**Files**: `packages/core/src/classifier/cue-scoring.ts`

**Changes**:
- Add an exported function `buildEnrichedCueText(baseSummary: string, baseExcerpt: string, followupAnswers?: readonly { field_target: string; answer: string | boolean }[]): string` that:
  - Returns `${baseSummary} ${baseExcerpt}` when no follow-up answers exist (preserves current behavior)
  - Appends serialized follow-up answers as `\n${field_target}: ${answer}` lines when present, so answer tokens like "leak", "suite", "drain" enter the cue keyword matching corpus
- The function is pure (no side effects, no imports beyond the type) and co-located with `computeCueScores` since it's a cue-scoring concern

**Why here**: Placing the helper in `cue-scoring.ts` rather than `answer-followups.ts` keeps cue-text construction testable in isolation and co-located with the scorer that consumes it. The handler just calls `buildEnrichedCueText(...)` and passes the result to `computeCueScores(...)`.

**Acceptance criteria**:
- [ ] `buildEnrichedCueText('summary', 'excerpt')` returns `'summary excerpt'` (no follow-ups)
- [ ] `buildEnrichedCueText('summary', 'excerpt', [{field_target: 'Maintenance_Problem', answer: 'leak'}])` returns `'summary excerpt\nMaintenance_Problem: leak'`
- [ ] Boolean answers are stringified: `answer: true` → `'true'`
- [ ] Re-exported from `packages/core/src/classifier/index.ts` (alongside existing `computeCueScores` export). No addition needed in `packages/core/src/index.ts` — the only consumer is `answer-followups.ts`, which imports directly from the classifier barrel.

**Tests**:
- Add unit tests in `packages/core/src/__tests__/classifier/cue-scoring.test.ts` (or create it) covering the three acceptance criteria above

### Task 2: Wire enriched cue text into answer-followups handler

**Files**: `packages/core/src/orchestrator/action-handlers/answer-followups.ts`

**Changes**:
- Import `buildEnrichedCueText` from `../../classifier/cue-scoring.js` (matching existing classifier imports at `answer-followups.ts:8-21`)
- Replace line 152:
  ```typescript
  // Before (buggy):
  const cueScoreMap = computeCueScores(`${issue.summary} ${issue.raw_excerpt}`, cueDict);

  // After:
  const enrichedText = buildEnrichedCueText(
    issue.summary,
    issue.raw_excerpt,
    issue.issue_id === targetIssueId ? followupAnswers : undefined,
  );
  const cueScoreMap = computeCueScores(enrichedText, cueDict);
  ```
- No changes to initial classification in `start-classification.ts:106` — follow-up answers don't exist at that point

**Why this works**: `cue_strength` has 40% weight in the confidence formula (`confidence.ts:49-72`). Enriching the scored text with answer tokens like "leak" and "drain" raises the maintenance cue score well above the 0.70 `category_gating_threshold`, which triggers category gating (`confidence.ts:247-275`) to prune cross-domain fields from `fieldsNeedingInput`. The classifier LLM already receives follow-up answers in its prompt (`classifier-prompt.ts:264-269`) — this fix aligns the cue-confidence path with the LLM path.

**Acceptance criteria**:
- [ ] After wiring, a generic opener + maintenance follow-up answers no longer leaves `Category` in `fieldsNeedingInput` (the behavioral guarantee — independent of the exact numeric threshold)
- [ ] Management-side fields (`Management_Category`, `Management_Object`) are pruned from `fieldsNeedingInput` via category gating
- [ ] Non-target issues in the same conversation still use un-enriched text (preserving current behavior)
- [ ] All existing `answer-followups.test.ts` tests still pass

**Tests**:
- Existing test suite passes without modification (enrichment is additive — tests that already had high cue coverage are unaffected)

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Enriched cue text over-boosts confidence on ambiguous cases (e.g., tenant answer matches multiple labels) | Low | Cue scoring already handles ambiguity via the `ambiguity` penalty in the confidence formula. The enrichment adds signal, not noise — follow-up answers are tenant-confirmed evidence. |
| `display_fields` breaks consumers that expect only `classification` | Low | `display_fields` is additive and optional. Existing `classification` record is preserved unchanged. Component falls back to chips when `display_fields` is absent. |
| Field labels in `taxonomy-labels.json` drift from taxonomy field additions | Low | `getFieldLabel` has a fallback for unmapped fields. Add a note in `taxonomy.ts` to update labels when adding fields. |
| Cue keyword matching on `field_target:` prefix text produces spurious hits | Negligible | Cue dictionaries contain domain terms ("leak", "drain"), not schema field names ("Maintenance_Problem"). The prefix is inert from a matching perspective. |

## Assumptions

- Chosen scope is the focused fix path for BUG-004, not the broader confidence/follow-up overhaul (BUG-005, BUG-007).
- Chosen confirmation treatment is a fielded summary, not a prose sentence summary.
- Tenant-facing confirmation hides cross-domain `not_applicable` values instead of showing muted "N/A".
- `ConfirmationPayload` remains TypeScript-only (no JSON Schema addition required).
- Broader follow-up UX policy work (BUG-005) and confidence-calibration work (BUG-007) remain separate downstream plans.
