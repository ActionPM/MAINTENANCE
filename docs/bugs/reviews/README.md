# Bug Portfolio Reviews

Store dated bug portfolio review outputs here.

Rules:

- Use one file per review date, for example `2026-03-18-bug-portfolio-review.md`.
- Treat each file as a point-in-time snapshot of the backlog and architecture review.
- Do not use these files as the living status tracker.
- Do not overwrite prior reviews to represent current state.
- Keep the living state in `docs/bug-tracker.md`.

Typical review contents:

- open bug clusters,
- duplicate candidates,
- subsystem themes,
- shared architectural causes,
- and proposed downstream plans, RFCs, tracker updates, or eval work.
