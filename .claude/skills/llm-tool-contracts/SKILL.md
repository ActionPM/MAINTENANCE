# LLM Tool Contracts

You are implementing one of the three bounded LLM tools for the **Service Request Intake & Triage Agent**. These tools are the ONLY interface between the LLM and the system. Every rule below is non-negotiable.

---

## Governing Principle — Governed Agent

The LLM proposes; deterministic code enforces. The model is a worker behind a schema wall — it never touches state, events, notifications, or escalation directly. The orchestrator calls the tool, validates the output, and decides what happens next.

---

## Ordering Rule — Split first, always (spec §2 item 2, §13)

```
IssueSplitter → (tenant confirms split) → IssueClassifier → FollowUpGenerator (if needed)
```

**The classifier CANNOT run until the split is finalized.** This is enforced at two levels:

1. **State machine**: `START_CLASSIFICATION` is only valid from state `split_finalized`
2. **Orchestrator guard**: before calling IssueClassifier, assert `conversation.state === 'split_finalized'` or `conversation.state === 'classification_in_progress'` (re-entry from follow-ups)

There are no shortcuts. A single-issue message still goes through the splitter (which returns one issue). Never skip the split step.

---

## The Three Tools

### 1. IssueSplitter

**Purpose:** Takes raw tenant text and returns a structured list of discrete issues.

**When called:** After `SUBMIT_INITIAL_MESSAGE`, conversation transitions to `split_in_progress`.

**Input contract:**

```typescript
interface IssueSplitterInput {
  raw_text: string; // tenant's message (max 8000 chars, already sanitized)
  conversation_id: string;
  taxonomy_version: string; // pinned version
  model_id: string; // pinned model
  prompt_version: string; // pinned prompt
}
```

**Output contract (must validate against `issue_split.schema.json`):**

```typescript
interface IssueSplitterOutput {
  issues: Array<{
    issue_id: string; // generated by the tool or orchestrator
    summary: string; // concise description (max 500 chars)
    raw_excerpt: string; // the portion of raw_text this issue came from
  }>;
  issue_count: number; // must equal issues.length
}
```

**Validation rules:**

- `issues.length` must equal `issue_count`
- `issues.length` must be >= 1 and <= 10
- Each `summary` must be <= 500 chars
- Each `raw_excerpt` must be a substring or close paraphrase of `raw_text`
- No issue may have an empty `summary`

**Post-validation:** Orchestrator transitions to `split_proposed` and presents the split to the tenant for confirmation.

---

### 2. IssueClassifier

**Purpose:** Takes ONE finalized issue and returns taxonomy classifications with per-field confidence.

**When called:** After split is finalized, orchestrator calls this once per issue. Conversation is in `classification_in_progress`.

**Precondition — ENFORCED:** `conversation.state === 'split_finalized'` (initial) or `classification_in_progress` (re-entry from follow-ups). Reject any call where split is not finalized.

**Input contract:**

```typescript
interface IssueClassifierInput {
  issue_id: string;
  issue_summary: string; // from finalized split
  raw_excerpt: string; // from finalized split
  followup_answers?: Array<{
    // from previous follow-up rounds (if re-classifying)
    field_target: string;
    answer: string | boolean;
  }>;
  taxonomy_version: string; // pinned
  model_id: string; // pinned
  prompt_version: string; // pinned
  cue_scores?: Record<string, number>; // pre-computed cue_strength per field (from cue dict)
}
```

**Output contract (must validate against classification schema):**

```typescript
interface IssueClassifierOutput {
  issue_id: string;
  classification: Record<string, string>; // field_name → taxonomy enum value
  model_confidence: Record<string, number>; // field_name → raw model hint [0..1]
  missing_fields: string[]; // fields the model could not determine
  needs_human_triage: boolean;
}
```

**Validation rules — three layers:**

1. **Schema validation:** All fields present, correct types, enum values exist in `taxonomy.json`
2. **Domain validation — category gating (spec §5.3):**
   - If top-level category is "management" but maintenance-specific fields are populated → contradictory
   - If top-level category is "maintenance" but management-specific fields are populated → contradictory
   - Contradictory = classification validation failure (see retry logic below)
3. **Confidence computation:** Applied after validation passes (see §Confidence Heuristic below)

---

### 3. FollowUpGenerator

**Purpose:** Given classification gaps (low/medium confidence fields, missing fields), generates targeted questions for the tenant.

**When called:** After classification, if any fields need tenant input. Conversation is in `needs_tenant_input`.

**Input contract (must validate against `followup_request.schema.json`):**

```typescript
interface FollowUpGeneratorInput {
  issue_id: string;
  classification: Record<string, string>; // current best classification
  confidence_by_field: Record<string, number>; // computed confidence per field
  missing_fields: string[];
  fields_needing_input: string[]; // fields with confidence < threshold
  previous_questions: Array<{
    // to avoid re-asking
    field_target: string;
    times_asked: number;
  }>;
  turn_number: number; // current turn (1-indexed)
  total_questions_asked: number; // running total across all turns
  taxonomy_version: string;
  prompt_version: string;
}
```

**Output contract (must validate against `followups.schema.json`):**

```typescript
interface FollowUpGeneratorOutput {
  questions: Array<{
    question_id: string;
    field_target: string; // one question → one field, always
    prompt: string; // human-readable question text
    options: string[]; // quick-reply options (preferred over free text)
    answer_type: 'enum' | 'yes_no' | 'text';
  }>;
}
```

**Hard caps — enforce in code BEFORE sending to LLM and AFTER receiving output:**

| Cap                | Limit | Enforcement                                                     |
| ------------------ | ----- | --------------------------------------------------------------- |
| Questions per turn | 3     | Truncate output if model returns more                           |
| Total turns        | 8     | Do not call FollowUpGenerator if `turn_number > 8`              |
| Total questions    | 9     | Do not call if `total_questions_asked >= 9`                     |
| Re-asks per field  | 2     | Exclude field from `fields_needing_input` if `times_asked >= 2` |

**When caps are exhausted and fields are still incomplete:**
→ Create WO with `needs_human_triage = true`, `missing_fields` preserved.
→ This is the escape hatch — do not loop forever.

**Event recording:** Every follow-up turn MUST be recorded in `followup_events` using the schema from spec §7.1.

## The Validation Pipeline (applies to ALL three tools)

Every LLM tool call follows the same pipeline. No exceptions. No shortcuts for "simple" calls.

```
┌─────────────┐
│  LLM Call    │
└──────┬──────┘
       ▼
┌─────────────┐     ┌──────────────────┐
│ JSON.parse  │────▶│ Parse failure     │
└──────┬──────┘     │ → retry(1x) with  │
       │            │   tighter prompt   │
       ▼            └────────┬──────────┘
┌─────────────┐              │ still fails
│  Schema     │              ▼
│  Validate   │────▶ LLM_FAIL (retryable or terminal)
└──────┬──────┘
       │            ┌──────────────────┐
       ▼            │ Schema failure    │
┌─────────────┐     │ → retry(1x) with  │
│  Domain     │     │   error context   │
│  Validate   │     └────────┬──────────┘
└──────┬──────┘              │ still fails
       │                     ▼
       │              LLM_FAIL (retryable or terminal)
       ▼
┌─────────────┐     ┌──────────────────────────────┐
│  Domain     │────▶│ Domain failure (contradictory)│
│  Pass?      │     │ → ONE constrained retry       │
└──────┬──────┘     │   with hard constraint to set │
       │            │   irrelevant fields to N/A     │
       ▼            └────────┬─────────────────────┘
   ✅ ACCEPT                 │ still contradictory
                             ▼
                    needs_human_triage = true
                    Store conflicting outputs in
                    classification_events as audit
```

### Implementation:

```typescript
async function callLLMTool<TInput, TOutput>(
  tool: LLMTool<TInput, TOutput>,
  input: TInput,
  outputSchema: JSONSchema,
  domainValidator: (output: TOutput) => DomainResult,
): Promise<ToolResult<TOutput>> {
  // Attempt 1
  const raw = await tool.call(input);

  // Parse
  let parsed: unknown;
  try {
    parsed = JSON.parse(raw);
  } catch {
    // Retry with tighter prompt
    const retryRaw = await tool.call(input, { retryHint: 'respond_only_valid_json' });
    try {
      parsed = JSON.parse(retryRaw);
    } catch {
      return { status: 'llm_fail', retryable: true, error: 'parse_failure_after_retry' };
    }
  }

  // Schema validate
  const schemaResult = schemaValidate(parsed, outputSchema);
  if (!schemaResult.valid) {
    // Retry with error context
    const retryRaw = await tool.call(input, {
      retryHint: 'schema_errors',
      errors: schemaResult.errors,
    });
    const retryParsed = safeParse(retryRaw);
    if (!retryParsed)
      return { status: 'llm_fail', retryable: true, error: 'parse_failure_on_schema_retry' };
    const retrySchema = schemaValidate(retryParsed, outputSchema);
    if (!retrySchema.valid) {
      return {
        status: 'llm_fail',
        retryable: false,
        error: 'schema_failure_after_retry',
        details: retrySchema.errors,
      };
    }
    parsed = retryParsed;
  }

  const output = parsed as TOutput;

  // Domain validate
  const domainResult = domainValidator(output);
  if (!domainResult.valid) {
    // ONE constrained retry
    const constrainedRaw = await tool.call(input, {
      retryHint: 'domain_constraint',
      constraint: domainResult.constraint, // e.g., "set maintenance fields to N/A"
    });
    const constrainedParsed = safeParse(constrainedRaw);
    if (!constrainedParsed) return { status: 'needs_human_triage', conflicting: [output] };
    const constrainedSchema = schemaValidate(constrainedParsed, outputSchema);
    if (!constrainedSchema.valid) return { status: 'needs_human_triage', conflicting: [output] };
    const constrainedDomain = domainValidator(constrainedParsed as TOutput);
    if (!constrainedDomain.valid) {
      // Store BOTH attempts in audit
      return { status: 'needs_human_triage', conflicting: [output, constrainedParsed as TOutput] };
    }
    return { status: 'ok', output: constrainedParsed as TOutput };
  }

  return { status: 'ok', output };
}
```

### Key rules:

- **One retry per failure type** — not one retry total. Parse failure gets one retry. Schema failure gets one retry. Domain failure gets one constrained retry.
- **Retries inject context** — the retry prompt tells the model what went wrong and how to fix it.
- **After all retries exhausted** → `LLM_FAIL` (triggers `llm_error_retryable` or `llm_error_terminal` in state machine) or `needs_human_triage` (for domain failures where we can still create a WO).
- **Never silently accept invalid output.** Not even once. Not even for "simple" single-issue splits.

## Confidence Heuristic (spec §14.3)

This is computed AFTER the classifier output passes schema and domain validation. It is deterministic code, not an LLM judgment.

### Formula (per field):

```
conf = clamp01(
    0.40 * cue_strength
  + 0.25 * completeness
  + 0.20 * model_hint
  - 0.10 * disagreement
  - 0.05 * ambiguity_penalty
)
```

### Input variables:

| Variable            | Source                                                        | Range                      | Notes                                                 |
| ------------------- | ------------------------------------------------------------- | -------------------------- | ----------------------------------------------------- |
| `cue_strength`      | Cue dictionary scoring (see below)                            | 0..1                       | Keyword/regex match score for the top candidate label |
| `completeness`      | How many of this field's required inputs are present          | 0..1                       | 1.0 if all context available, lower if info missing   |
| `model_hint`        | `IssueClassifierOutput.model_confidence[field]`               | **clamped to [0.2, 0.95]** | Raw model confidence clamped before use               |
| `disagreement`      | Whether cue_strength and model_hint point to different labels | 0 or 1                     | 1 if cue top label != model top label                 |
| `ambiguity_penalty` | Whether multiple labels score similarly                       | 0..1                       | Higher when top-2 labels are close in score           |

### Model hint clamping (critical):

```typescript
const clampedModelHint = Math.max(0.2, Math.min(0.95, rawModelConfidence));
```

The model is never allowed to express 100% or near-0% confidence. This prevents overconfident or underconfident model outputs from dominating the heuristic.

### clamp01:

```typescript
function clamp01(x: number): number {
  return Math.max(0, Math.min(1, x));
}
```

### Threshold bands:

| Band       | Range       | Action                                                     |
| ---------- | ----------- | ---------------------------------------------------------- |
| **High**   | >= 0.85     | Accept — no follow-up needed for this field                |
| **Medium** | 0.65 – 0.84 | Ask if the field is required or risk-relevant              |
| **Low**    | < 0.65      | Must ask (unless field is not applicable to this category) |

These bands determine which fields go into `fields_needing_input` for the FollowUpGenerator.

**Bands are provisional** — validate and tune on Gold Set B before pilot launch.

## Cue Dictionary (spec §14.4)

### File: `packages/schemas/classification_cues.json`

This is a required artifact. It defines per-field keyword and regex cues used to compute `cue_strength`.

### Structure:

```json
{
  "version": "1.0.0",
  "fields": {
    "<taxonomy_field_name>": {
      "<label_value>": {
        "keywords": ["word1", "word2", "..."],
        "regex": ["pattern1", "pattern2"]
      }
    }
  }
}
```

### Example:

```json
{
  "version": "1.0.0",
  "fields": {
    "maintenance_category": {
      "plumbing": { "keywords": ["leak", "toilet", "sink", "drain", "pipe", "water"], "regex": [] },
      "electrical": { "keywords": ["breaker", "outlet", "switch", "sparks", "power"], "regex": [] },
      "hvac": { "keywords": ["heat", "ac", "furnace", "thermostat", "vent"], "regex": [] }
    },
    "maintenance_object": {
      "toilet": { "keywords": ["toilet", "wc", "commode"], "regex": [] },
      "faucet": { "keywords": ["faucet", "tap", "spigot"], "regex": [] }
    }
  }
}
```

### Scoring algorithm:

```typescript
function computeCueStrength(text: string, fieldName: string, cueDict: CueDictionary): number {
  const fieldCues = cueDict.fields[fieldName];
  if (!fieldCues) return 0;

  const scores: Array<{ label: string; score: number }> = [];

  for (const [label, cues] of Object.entries(fieldCues)) {
    let hits = 0;
    const totalCues = cues.keywords.length + cues.regex.length;
    if (totalCues === 0) continue;

    const lowerText = text.toLowerCase();

    // Keyword hits
    for (const keyword of cues.keywords) {
      if (lowerText.includes(keyword.toLowerCase())) hits++;
    }

    // Regex hits
    for (const pattern of cues.regex) {
      if (new RegExp(pattern, 'i').test(text)) hits++;
    }

    scores.push({ label, score: hits / totalCues }); // normalized 0..1
  }

  // cue_strength = top score across all candidate labels
  if (scores.length === 0) return 0;
  scores.sort((a, b) => b.score - a.score);
  return scores[0].score;
}
```

**Rules:**

- Cue dictionary labels MUST match `taxonomy.json` values exactly
- If a field has no cues defined, `cue_strength = 0` (model_hint carries more weight)
- Cue strength is computed BEFORE calling the classifier and passed as input context
- Also used post-classification to detect disagreement (cue top label vs model top label)

## Category Gating — The Detailed Error Path (spec §5.3)

This is the most nuanced domain validation. Get it right.

### What is contradictory:

- Classifier says top-level category is **management** but populated maintenance-specific fields (maintenance_category, maintenance_object, etc.)
- Classifier says top-level category is **maintenance** but populated management-specific fields

### Step-by-step:

1. Classifier output passes schema validation ✅
2. Run category gating check: are cross-domain fields populated?
3. **If contradictory:**
   a. This is a **classification validation failure** — not a schema failure
   b. Run ONE targeted retry with hard constraint:
   - Inject instruction: "The top-level category is [X]. Set all [other domain] fields to `not_applicable`."
     c. Validate the retry output (full pipeline: parse → schema → domain)
     d. **If still contradictory after retry:**
   - Set `needs_human_triage = true`
   - Store BOTH classification attempts (original + retry) in `classification_events` as audit records
   - Proceed via escape hatch: create WO with status `created`, `missing_fields` preserved
   - Do NOT block the tenant — they get their WO, it just needs human review

### What "not-applicable equivalents" means:

- Use the literal value `"not_applicable"` for each irrelevant domain field
- This value MUST exist in `taxonomy.json` for every domain-specific field
- Do NOT use `"other_*"` values as N/A — those mean "this domain applies but doesn't fit a named category"
- Do NOT use null, empty string, or omission as N/A
- This is enforced by the domain validator (taxonomy-cross-validator), NOT by JSON Schema — classifier output schemas type values as generic strings

## Version Pinning (spec §5.2)

Every LLM tool call MUST use the conversation's pinned versions:

```typescript
const versions = conversation.pinned_versions;
// {
//   taxonomy_version: "1.2.0",
//   schema_version: "1.0.0",
//   model_id: "claude-sonnet-4-6",
//   prompt_version: "2.1.0"
// }

// Pass to every tool call
const input: IssueClassifierInput = {
  ...issueData,
  taxonomy_version: versions.taxonomy_version,
  model_id: versions.model_id,
  prompt_version: versions.prompt_version,
};
```

**Resumed conversations retain their pinned versions** even if newer versions exist. Never silently upgrade mid-conversation.

**Validation output must also record versions** — every `classification_events` row stores the `taxonomy_version`, `model_id`, and `prompt_version` used.

## Tool Isolation Rules

1. **Only the orchestrator calls LLM tools.** No endpoint handler, UI component, or service may invoke IssueSplitter, IssueClassifier, or FollowUpGenerator directly.

2. **Tools have no side effects.** They take input, return output. They do not:
   - Write to the database
   - Send notifications
   - Transition conversation state
   - Call other tools
     The orchestrator handles all of that after validating the output.

3. **Tools are stateless.** Each call is independent. If context from previous rounds is needed (e.g., follow-up answers for re-classification), the orchestrator assembles it and passes it in the input.

4. **Tools are replaceable.** The orchestrator depends on the input/output contracts, not the implementation. Swapping the underlying model or prompt is a configuration change, not a code change (enabled by version pinning).

## Implementation Checklist

For every LLM tool implementation or modification:

```
[ ] Output validates against its JSON Schema in packages/schemas/
[ ] Full validation pipeline wired: parse → schema → domain → accept/retry/fail
[ ] Parse failure: one retry with tighter prompt
[ ] Schema failure: one retry with error context injected
[ ] Domain failure: one constrained retry, then needs_human_triage
[ ] Both original and retry outputs stored in events when needs_human_triage
[ ] Confidence heuristic computed in deterministic code (not by the model)
[ ] Model hint clamped to [0.2, 0.95] before use in formula
[ ] Cue strength computed from classification_cues.json before classification call
[ ] Disagreement flag set when cue top label != model top label
[ ] Follow-up hard caps enforced in code: 3/turn, 8 turns, 9 total, 2 re-asks/field
[ ] Escape hatch: WO created with needs_human_triage if caps exhausted
[ ] Every follow-up turn recorded in followup_events (§7.1 schema)
[ ] Version pins passed to every tool call from conversation.pinned_versions
[ ] Classification and follow-up events record the versions used
[ ] Tool has no side effects — orchestrator handles state, events, notifications
[ ] Split-first ordering enforced: classifier rejects if split not finalized
[ ] Category gating check runs on every classification output
[ ] Taxonomy enum values come from taxonomy.json, never hardcoded
```
